1. Méthodologie

Sur un projet à un seul développeur de cette taille, j'ai retenu une approche Kanban simplifié avec une découpe en blocs fonctionnels. C'est la méthode la plus pragmatique :

  • Pas de cérémonies inutiles (un sprint = un développeur n'a pas de sens).
  • Visibilité immédiate de ce qui est en cours / fait.
  • Possibilité de réordonner les priorités sans rituel.

Le tableau Kanban est tenu dans Notion. Les colonnes : Backlog → À faire → En cours → Revue → Fait.

4. Gitflow appliqué

main      ───●────────●────────●─────────── (production)
              \        \        \
develop    ────●────●───●────●───●──────────── (intégration)
                \    \    \    \    \
features        ●    ●    ●    ●    ●     (1 branche = 1 feature)

Règles appliquées :

  • Une branche par fonctionnalité nommée feature/<nom>, issue de develop.
  • Après tests, merge vers develop avec message conventionnel (feat:, fix:, docs:, chore:, refactor:).
  • Quand develop est stable, merge vers main (release).
  • Pas de commit direct sur main.

Branches créées dans ce projet :

  • main (release stable)
  • develop (intégration)
  • feature/database-schema (schéma BDD + scaffolds + features)

5. Communication & revue

Dans un projet réel d'équipe :

  • Daily standup de 10 minutes (qui fait quoi, blocages).
  • Pull Requests avec au moins une review avant merge dans develop.
  • Tests d'intégration automatiques en CI (GitHub Actions).
  • Démos hebdomadaires avec le chef de projet pour cadrer le delta.

Ici, en projet solo, l'équivalent est :

  • Tâche commencée → marquée En cours dans Notion.
  • Tâche finie → branche mergée + commit conventionnel + colonne Done.
  • Tests manuels (parcours visiteur, user, employé, admin) après chaque feature majeure.
